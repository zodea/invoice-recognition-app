# ADR-0004：引入 vue-router + pinia（增量迁移）

日期：2026-06-16　状态：已采纳

## 背景

当前无前端路由：App.vue 用 reka-ui `Tabs` 切 4 个顶层视图（发票/送货单/分供方/单价对比），状态存于手写 `reactive` 单例（`src/store.js` 送货单、`src/invoiceStore.js` 发票）+ localStorage（`supplierDb.v1`、`priceCompareManualGroups`）。问题：

- **分供方详情是模态弹窗**，详情里再点编辑＝模态叠模态，嵌套深、体验差（issue #18）。
- 规划中要加**工地详情页**，识别明细库又要**按工地拆成多个 JSON**（ADR-0003 决策 6），需要一个能跨视图集中读写/装配的状态层。
- 跨视图共享数据多：工地/分区、分供方库、识别明细——手写单例传递渐繁。

## 决策

1. **引入 vue-router（正式路由页）+ pinia（集中状态）**。分供方详情：模态 → 路由页 `/supplier/:id`；新增**工地详情** `/site/:id`。

2. **增量迁移，不大爆炸**：先迁"**分供方 / 工地**"这条线（路由 + pinia store + 识别明细库装配）；**送货单/发票暂留** `store.js` / `invoiceStore.js`（已工作、切 tab 不丢），后续分批迁。

3. **顶栏 Tabs 与路由同步**：保留现有 4 tab 观感，`activeTab ↔ route` 双向绑定，避免一次性重做导航。

4. **识别明细库的"每工地 JSON"由 pinia 装配**：pinia 在内存汇总所有 `识别明细库/<工地>.json`，供「分供方详情」「单价对比」跨工地读、「工地详情」单工地读。

## 备选方案与否决理由

- **一次性全迁（big-bang）**：store.js / invoiceStore.js / supplier-db / price-compare 全进 pinia、顶栏改 router 驱动。最统一但一次改面大、回归风险高。否决为首选，留作后续收尾。
- **不引依赖、保持手写单例 + master-detail 整屏切换**：能去模态嵌套，但满足不了"正式路由（可刷新直达/前进后退）+ 集中共享状态 + 将来工地详情页"的诉求。否决。

## 影响

- 新增依赖 `vue-router`、`pinia`。
- **迁移期两套状态并存**（pinia 新线 + 旧单例），边界须清晰、避免双写同一份数据。
- 是 issue #14（识别明细库装配）、#18（分供方详情路由化 + 瀑布流）的**前置基础**，应先落地。
