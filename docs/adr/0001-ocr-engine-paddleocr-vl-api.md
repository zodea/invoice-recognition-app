# ADR-0001：手写送货单识别采用 PaddleOCR-VL 云 API，本地小模型仅作兜底

日期：2026-06-13　状态：已采纳

## 背景

送货单是手写+表格+红章的扫描件。对 6 家供应商真实扫描件的批量实测（2026-06-13）：
内置本地小模型（PP-OCRv2，paddlejs WebGL）5 家中 4 家输出 0 行文字（横躺方向检不出 + 算力弱），
密集两联单单页 102 秒超时；唯一有输出的一家也只能出印刷体（公司名/单号），手写明细全乱码。
用户核心诉求（识别率大幅提升、识别出材料明细、一页多单拆分）在纯本地路线下无法满足。

## 决策

识别主引擎采用 **PaddleOCR-VL-1.6**（百度 2026-05 开源的 0.9B 文档解析视觉大模型，
OmniDocBench 96.33% SOTA，手写/表格/印章显著强于传统 OCR），通过 **飞桨星河（AI Studio）
官方 API** 调用（端点 `…aistudio-hub.baidu.com/layout-parsing`，`Authorization: token <访问令牌>`）。
返回的 Markdown 表格直接解析成材料明细行；按表格切段实现"一页多单自动拆分"；
`useDocOrientationClassify` 解决方向问题。API 地址与令牌由用户在「设置」里配置，
仅存本机 localStorage，**绝不写入代码仓库**。

本地 PP-OCRv2（模型已随应用内置于 public/ocr-models，见 memory/ADR 外的实现说明）保留为
**离线兜底**：云调用失败时自动回退，并加了 0 行时旋转 90/270 重试。

## 备选方案与否决理由

- **纯本地优化（切块/预处理/方向重试）**：免费离线，但手写明细识别天花板低、低配机速度不可用——核心诉求满足不了。
- **本地部署 VL（vLLM/CPU 版）**：需要 NVIDIA GPU 或重型 Python 环境，用户的办公机不具备，且与"拷到别的电脑直接用"冲突。
- **传统云 OCR 产品（百度智能云通用/手写识别）**：可行，但表格结构还原弱于 VL（需要自己拼行列），且用户明确指定了 PaddleOCR-VL-1.6。

## 影响

- 识别质量依赖网络与星河平台可用性；离线时自动降级为本地小模型（仅印刷体可用）。
- 单据图片会上传到百度星河平台处理（用户已知情同意）。
- 免费额度内零成本；超量按平台计费。
- 多单拆分、明细行解析的实现都建立在 VL 的结构化输出上（src/lib/vl-parse.js）。
